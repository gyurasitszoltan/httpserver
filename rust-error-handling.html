Rust hibakezelés

Gyakorlati bevezető Result, Option, ?, saját hibatípusok és Axum HTTP-válaszok használatához.

1. A Rust nem exceptionökre épít

A várható hibákat a függvény visszatérési értéke jelzi. Egy függvény nem „eldob” egy exceptiont, hanem Err értékkel tér vissza.

fn parse_age(input: &str) -> Result<u8, String> {
    input.parse().map_err(|_| "Érvénytelen életkor".into())
}

A hívó két esetet kezel:

match parse_age("42") {
    Ok(age) => println!("Életkor: {age}"),
    Err(message) => println!("Hiba: {message}"),
}

2. Result<T, E>

A Result vagy sikeres érték (Ok(T)), vagy hiba (Err(E)).

VáltozatJelentés
Ok(value)A művelet sikerült, az eredmény value.
Err(error)A művelet nem sikerült, az ok error.
 
fn email_of(value: &str) -> Result<String, ApiError> {
    let email = value.trim().to_lowercase();
    if !email.contains('@') {
        return Err(ApiError::Bad("Érvényes e-mail cím szükséges.".into()));
    }
    Ok(email)
}

3. Mit csinál a ? operátor?

A ? az előtte álló Result értékét kezeli:

  • Ok(value) esetén kibontja a value-t;
  • Err(error) esetén azonnal visszatér az aktuális függvényből a hibával.
fn create(email_input: &str) -> Result<String, ApiError> {
    let email = email_of(email_input)?;
    // Ez a sor csak érvényes e-mailnél fut le.
    Ok(email)
}

Ugyanez kézzel:

fn create(email_input: &str) -> Result<String, ApiError> {
    let email = match email_of(email_input) {
        Ok(value) => value,
        Err(error) => return Err(error),
    };
    Ok(email)
}
Fontos: ez korai visszatérés, nem exception. A vezérlés nem „ugrik át” tetszőleges hívási szinteket; minden köztes függvénynek Result-ot kell visszaadnia és tovább kell adnia a hibát.

4. A jelenlegi user-create folyamat

let result = users::insert(
    &state.db,
    email_of(&request.email)?,
    name(request.display_name)?,
    role(&request.role)?,
    request.is_active.unwrap_or(true),
).await;

A paraméterkifejezések kiértékelődnek. Bármelyik ? hibánál azonnal kilép a create handlerből:

Err(E::Bad(...))  →  create() visszaadja az Err-t
                  →  Axum meghívja az E::into_response()-t
                  →  HTTP 400 + JSON hibaválasz

5. Option<T>: amikor nem hiba a hiány

Az Option kétféle állapotot ír le: Some(value) vagy None. Például egy nem létező adatbázisrekord gyakran nem szerverhiba.

let user = users::find(&db, id).await?;
let user = user.ok_or(ApiError::NotFound)?;

Az első ? az adatbázishibát továbbítja. A második sor a normál None értéket tudatosan 404 Not Found hibává alakítja.

6. Saját hibatípus

Alkalmazásban érdemes egy közös, beszédes hibatípust használni. A thiserror crate leveszi az alap boilerplate-et.

use thiserror::Error;

#[derive(Debug, Error)]
enum ApiError {
    #[error("{0}")]
    Bad(String),
    #[error("unauthorized")]
    Unauthorized,
    #[error("not found")]
    NotFound,
    #[error("database error: {0}")]
    Database(#[from] sqlx::Error),
}

A #[from] miatt egy sqlx::Error automatikusan ApiError::Database-szé alakítható ? használatakor.

7. Hiba HTTP-válasszá alakítása Axumban

Az Axum IntoResponse traittel dönti el, hogyan legyen egy saját hiba HTTP-válasz.

impl IntoResponse for ApiError {
    fn into_response(self) -> Response {
        match self {
            ApiError::Bad(message) => (
                StatusCode::BAD_REQUEST,
                Json(json!({ "error": { "message": message } })),
            ).into_response(),
            ApiError::NotFound => StatusCode::NOT_FOUND.into_response(),
            _ => StatusCode::INTERNAL_SERVER_ERROR.into_response(),
        }
    }
}

Így a handler maradhat egyszerű:

async fn show(Path(id): Path<i64>) -> Result<Json<User>, ApiError> {
    let user = users::find(id).await?.ok_or(ApiError::NotFound)?;
    Ok(Json(user))
}

9. unwrap() és expect()

Ezek hibánál pánikot okoznak, ezért HTTP request feldolgozásakor általában ne használd őket.

// Rossz kérésből szerverpánik lehet:
let user = users::find(&db, id).await.unwrap();

// Jó: a hiba szabályozott HTTP választ eredményez:
let user = users::find(&db, id).await?;
Kivétel: unwrap() elfogadható lehet tesztben, vagy bizonyítottan konstans/belső értéknél. Külső inputnál és adatbázisnál ne használd.

10. Rövid szabályrendszer

  1. Várható hiba: Result.
  2. Normál hiányzó érték: Option.
  3. Egyszerű hibatovábbítás: ?.
  4. API-ban közös saját hibatípus és IntoResponse.
  5. Nyers belső hibát ne küldj a kliensnek; naplózd, a kliensnek adj általános üzenetet.
